# Nomi 项目工程纪律 — 给 Claude 的读取项

> 每个会话开始**强制读这份**。下面两条是用户多次明确强调的硬规则，违反 = 工作错误。

## 规则 1：加新必删旧（No Parallel Versions）

**当我引入一个新组件 / 新流程替代旧的：**

- 同一 commit 里**删除被替代的旧代码**
- 不留"逃生口 / Collapse / 备选 fallback / 高级开关"——这些都是借口
- 死代码（grep 不到外部引用）立刻物理删除文件，不留"以防万一"
- 反例：M5.3 我加了 `OnboardingWizard` 但留了 `ModelCatalogImportSection`，结果用户看到两套 UI 混杂。错。
- 正例：今天的 `chore: codebase cleanup (audit 2026-05-30 section A + C)` —— 移动文件同步更新所有 import，0 残余。

**如果"不敢删"：**
- 真的不确信新方案 → 不要 ship 新方案
- 怕回归 → 加测试，不是留旧代码
- 旧的有点价值 → 把那点价值合并到新代码里，再删旧

## 规则 2：用户视角 + 极简

**做 UI / 文案 / 卡片设计前，先按这个顺序问：**

1. **用户进来要看什么？** 列出每条信息，按重要度 ⭐⭐⭐ / ⭐⭐ / ⭐ 排序。
2. **每条信息有用户行动价值吗？** 没有行动价值的信息 = 噪音 = 删。
3. **0 权重的常见嫌疑犯：**
   - 节点功能描述文字（"承接上游提示词生成图片。"）
   - onboarding 文案在每次出现（"或选中后输入提示词生成"）
   - 重复的分组标签（节点已在 "角色" group 里，节点头还写"角色"）
   - 长 error stack 灌满卡片（应缩成 ⚠️ + tooltip）
   - 双层 border / 圆角套圆角（违反 Design.md 第 6 节）
   - 永远 80px 高的"信息区"即使内容为空
4. **设计系统**：参考 `Design.md` + `src/design/`，token-only，光模式，密度优先。

**反例**：M5.3 wizard 首版我写"高级 / 专家逃生口"折叠区，留着 600 行旧代码。错。
**正例**：今天角色卡删了"角色"tag、内层 border、"或选中..."文案、用户不需要的描述行。

## 规则 3：决策格式

涉及范围 / 取舍时，**先给用户用户视角的对比表**，再让用户拍板：

```
| 方案 | 用户看到什么 | 代价 |
|---|---|---|
| A | ... | ... |
| B | ... | ... |
```

不要单方面"我建议 A"然后开干。

## 规则 4：执行前必写文档

涉及多文件 / 多步骤的改动，先在 `docs/plan/` 或 `docs/audit/` 写一份执行文档，含：
- 范围
- 不动什么
- 回滚策略
- 验收门

写完用户能预读 / 反驳；执行完回填结果。

## 规则 5：碰框架 / 库，先查官方文档再写（Context7 强制）

> 反复出现的硬伤：AI 自己手搓一套，而不是用库的官方实现。例：手写 viewport
> transform，而 React Flow 早就有 `screenToFlowPosition` / `zoomOnScroll` /
> 受控 `viewport`。这种"看起来能跑、其实跟生态脱节"的代码 = bug 温床 + 维护债。

**铁律**：凡是涉及第三方框架 / 库的实现或改动（React Flow、AI SDK、Mantine、
Electron、Tiptap、Vite…），**动手写之前必须先用 `context7` 查该库的官方文档 / 推荐
用法**，再按官方实现来写。已配置 Context7 MCP（项目级 `.mcp.json`）。

流程：
1. `resolve-library-id` 拿到库的 Context7 ID（如 React Flow → `xyflow/xyflow`）。
2. `get-library-docs` 拉该库相关主题的官方文档（带上 topic，如 "viewport zoom"）。
3. 对照官方推荐的 API / 模式实现；**不要凭记忆或臆测手搓**。
4. 若官方就是没有现成能力，才允许自定义，并在注释里写明"官方无此能力，故自实现"。

**不查就写 = 工作错误**，等同违反规则 1（手搓的等于引入了一套与官方并行的劣质实现）。

## 规则 6：先研究顶尖开源项目的真实代码，再做方案（不重复造轮子）

> 规则 5 用 Context7 查"库的官方 API"；这条更上一层——查"**架构与思路**"。
> 反复出现的硬伤：我凭记忆/印象就出方案，而没去看业界顶尖项目**真实代码**怎么解的。

**铁律**：做任何项目方案前，先去 GitHub 找 1~N 个**顶尖开源项目**，读它们的**真实代码/实现**（不是只扫 README、不是凭印象），产出：
- 它们**怎么做的**（具体到模块/数据结构/流程，给出文件/代码位置）
- 我们**能直接借鉴什么**
- **哪里不适用**、为什么

能借鉴成熟实现就别重复造轮子。允许的"自创"只在确认无现成方案后，且写明理由。
本地 agent runtime 方向的参考池（示例，非穷尽）：Cline、OpenHands、Aider、Continue、
opencode、Cherry Studio、Jan、LobeChat；画布/节点向：tldraw、ComfyUI、xyflow。

## 规则 7：项目方案定稿前，必过 6 角色子 agent 审查

**触发条件**：任何涉及架构 / 取舍 / UX 的"项目方案"，在给用户拍板前。

**铁律**：用 Agent 工具创建子 agent，从以下 **6 个角色各审一遍**，再汇总：
1. **CTO** —— 架构合理性、可维护、技术债、扩展性、与现有内核是否冲突
2. **设计师** —— 视觉/一致性、是否对齐 `Design.md` 与 `src/design/`、密度优先
3. **产品经理** —— 用户价值、优先级、范围是否过大、是否解决真痛点
4. **前端** —— 实现可行性、状态管理、性能、组件复用
5. **后端** —— runtime / 数据 / 安全 / IPC / 持久化影响
6. **真实用户**（最终综合体验）—— **不只是代码，而是设计 + 体验 + 界面的综合体感**：
   "我作为用户，用起来顺不顺、爽不爽、看着舒不舒服"

**流程**：研究开源（规则 6）→ 起草方案 → 6 角色并行审查 → 汇总意见、修方案 →
带对比表（规则 3）给用户拍板。目的不是只验"代码对不对"，而是**综合体感过关**。

## 规则 8：用户可见的设计改动，先出"可视样张"再做

> 反复教训：直接动手做 UI，做完才发现方向/样子不对，返工。光文字描述方案，用户也想象不出最终长啥样。

**触发条件**：任何**用户会看到**的东西——界面 / 卡片 / 节点 / 面板 / 交互 / 文案排版。

**铁律（动手实现前必须走完）**：
1. **参考设计系统**：`Design.md` + `docs/design/nomi-design-system.md` + `src/design/`（token、现有组件），不凭空设计、不手搓重复组件。
2. **出"可视样张"——最低成本让用户先看到样子**：优先一张 **HTML mockup**（或最简化的真实实现 demo），而不是只有文字。用户能直接打开看到长相再决定。
3. **评审**：设计师 Agent + 真实用户 Agent（规则 7）过一遍。
4. **用户本人确认**后，才进入实现。

**不出样张就开干 = 工作错误。**

**复杂交互（如时间轴 图片轨/媒体轨）**：尤其不能"一次性出发就干"。要**分多轮**：技术选型/框架调研（用 Context7 查核心框架官方做法，规则 5/6）→ 梳理功能列表 → 样张 → 评审 → 补充调整 → 再定方案 → 才实现。持续迭代，别图快。

## 规则 9：写码前先想模块化 / 架构（别把东西堆在一处）

> 反复出现的硬伤：上来就往一个文件 / 一个组件 / 一个 store 里塞，越滚越大，最后所有东西
> 混在一处——改一处怕动全身，死代码没人敢删，关注点全纠缠。

**铁律**：任何涉及新模块 / 新组件 / 新流程，或往现有大文件继续加东西之前，**先停下来想清楚整体的模块化与架构**：
- 这块逻辑**该放哪一层**？（UI / 状态 / 领域逻辑 / runtime / 持久化——别让它们混在一个文件里）
- 关注点是否分离？渲染归渲染、状态归 store、领域逻辑归领域层、副作用归 runtime。
- 是不是又在往一个巨壳里塞？（反例：`BaseGenerationNode.tsx` 1354 行什么都管；`styles.css` 8867 行混了多代画布。**这就是没先想架构的代价**。）
- 新东西和现有内核的边界在哪？会不会引入第二份真相源 / 并行实现（连回规则 1）。

**架构决策必须过讨论 + 调研，不能一个人拍**：
1. **拉角色 agent 一起讨论**——模块化 / 架构问题至少过 **CTO + 前端 + 后端**三角（用 Agent 工具角色扮演，规则 7 的子集）：分层是否合理、状态放哪、IPC / 持久化 / 数据边界是否清晰、可维护可扩展。
2. **做调研、查官方文档**——涉及框架 / 技术栈选型，先用 Context7 查官方推荐架构（规则 5），再读顶尖开源真实代码怎么分层（规则 6），**确保整个框架 / 技术栈的用法没问题**，不是凭记忆手搓一套。
3. 有取舍时出对比表给用户拍板（规则 3）。

**不先想模块化就开写 = 工作错误**，等同违反规则 1——堆出来的巨壳和混在一起的代码，就是下一轮要还的债。

## 规则 10：样式一律 Tailwind 写组件上，CSS 文件只可减不可增

> 反复出现的硬伤：组件迁去 Tailwind 了，旧 CSS 却没删——堆成 `styles.css` 8867 行死代码，
> 分 Tier 1/2/3 才清完（约 8700 行全是没人消费的尸体）。这是规则 1「加新必删旧」最贵的反例。

**铁律（碰样式时）**：
- **新样式一律用 Tailwind utility 写在组件 `className` 上**（utility-first，对齐 Tailwind 官方）。
  不在全局 CSS 里新建组件类。
- **不用 `@apply` 把 utility 重新打包成类**（官方不推荐）；重复就**抽组件 / template partial**，
  样式和结构就近放一起。
- **CSS 文件分工固定，且只可减不可增**：
  - 设计 token → `src/theme/nomi-tokens.css`（`--nomi-*`）/ Tailwind `@theme`。
  - 全局 reset / `body`/`#root` / keyframes → `src/styles/index.css` 或 `tailwind.config.ts`。
  - 第三方 DOM 覆盖（Mantine 等）→ `src/styles/vendor-overrides.css`。
  - `src/styles/globals.css`（Tier 3 后的残余全局）：**只准删、不准加组件样式**。
- **迁一个组件到 Tailwind = 同一 commit 删它的旧 CSS**（连回规则 1）。迁完忘删 = 工作错误。
- 真要写 CSS（官方也允许的少数情况）：写进对应分工文件、**消费 token 变量**（`var(--nomi-*)`），
  不靠 `@apply`，并注释为什么 Tailwind 表达不了。

**自检**：动手往任何 `.css` 文件加规则前问一句——「这能不能用 Tailwind utility 写在组件上？」
能 → 就那么写。不能（token/全局/keyframes/vendor）→ 进对应分工文件。**别再让 CSS 文件长回去。**
