## 一句话总结

上轮（07-15）把「闭环回灌」落成了 GroundShot 的调度改法；**本轮的主角是「参考图归属」**——快手可灵系的 ReBind（07-16，四天新）用实验坐实了 Nomi 早就用大白话说过的那个痛点：「三张参考图模型分不清谁是谁→张冠李戴」的根子**一半在指令侧**——普通指令没写清「哪个属性来自哪张图」，而且**通用 MLLM 自由发挥时写不可靠**。它的解法思想（把 `Ref N` 标签嵌在指令的语义位置上做显式归属）不需要它家训练的模型，**正好落在 Nomi PR#41 刚合的 `@imageN` 引用底座上**。同时 07-15 两天连出两把新尺子（KeyFrame-Compass 关键帧忠实度 / VGIF-Score 指令跟随约束图），正补 Nomi 图生视频桥和 shotVerify 的量化缺口。

**本轮最该动的一件事：给 Nomi 的多参考 prompt 补「显式归属」层**——① 拆镜头规划师/AI 优化写多参考 prompt 时，用代码级模板强制把每个实体绑到 `@imageN`（ReBind 实证：这事不能信 LLM 自由发挥）；② 多参考任务提交时 prompt 里若零绑定，按连线顺序 + 参考槽语义自动补一段归属前言。落点 `promptMentions.ts`（@imageN 单一真相源已建好）+ `storyboardPlan.ts`（规划师写 prompt 处）。第二件（次优先）：把 KeyFrame-Compass 的「首帧忠实度」轴接进图转视频桥的 shotVerify——Nomi image-first 模式整条链路押在「首帧图能约束住视频」上，这个假设现在有尺子可量了。诚实边界见各条。

## 🟢 可落地（这周/这月能进 Nomi）

### ReBind：多参考「张冠李戴」的根子一半在指令——把参考标签嵌在语义位置上做显式归属，思想可直接进 Nomi prompt 构造层（2026-07-16，四天新）

[arxiv 2607.14681](https://arxiv.org/abs/2607.14681) · [项目页 rebind-mrv2v.github.io](https://rebind-mrv2v.github.io/)（本轮实测 200，含 Code 链接指向 github.com/rebind-project/rebind，仓内容未逐项核）· 作者含 Pengfei Wan / Yuanxing Zhang / Xin Tao 同族作者群（快手可灵系特征，未在页面逐一确认）+ Wenhan Luo / Yike Guo（HKUST）· 日期：2026-07-16（本轮窗内最新可落地）。

- **它坐实的失败模式就是 Nomi 的老痛点**：多参考图条件生成里，「existing editing instructions lack explicit reference relationships, and most MLLMs cannot generate them reliably」——普通指令只描述画面、不写「这件外套来自图 2、这个人来自图 1」，模型就自己猜归属→张冠李戴。Nomi 用户决策记录里的原话例子（「三张参考图模型分不清谁是谁」）第一次有论文级实验背书，且指认了两个责任层：**指令格式**（没写归属）+ **写指令的 LLM**（自由发挥写不可靠）。
- **机制三件套（前两件是它家训练的模型，Nomi 不抄；第三件是格式思想，Nomi 直接用）**：① ReBind-Instruct——两阶段训练（结构化指令学习 + RL 归属优化）的专用 MLLM，产出带归属的指令；② ReBind-Edit——轻量适配 T2V 做 cross-attention binding；③ **中间表示格式**——「semantic instructions with embedded reference tokens」，项目页示例即自然语言里内嵌 `Ref 1 / Ref 2` 标签，嵌在**属性被提到的那个语义位置**（不是列在开头当注释）。
- **落到哪个文件 + 怎么改**（Nomi 底座已在，PR#41 2026-07-19 刚合）：
  - `src/workbench/assets/promptMentions.ts` — PR#41 建好的单一真相源：同一有序数组既产 chip 又把 `@[asset:url]` 投影成 `@imageN`（`projectPromptForSend`，严格按实际参考图顺序）。ReBind 落地不用新造词汇表，就用 `@imageN`。
  - `src/workbench/generationCanvas/runner/catalogTaskActions.ts:122` — 发送前最终投影点。在这里加**兜底**：多参考（≥2 张）任务的 prompt 里若一个 `@imageN` 绑定都没有，按连线顺序 + `referenceSlots.ts` 的槽语义（首帧/角色/场景）自动补一段归属前言（如「@image1=主角外观，@image2=场景」），比裸发少歧义。
  - `src/workbench/generationCanvas/agent/storyboardPlan.ts` + AI 优化节点的 prompt 模板 — **主改动**：规划师/优化器给带多参考的镜写 prompt 时，模板里强制要求每个 recurring 实体在语义位置绑 `@imageN`；产出后用**代码级校验**（正则查「参考数 N ≥2 但 prompt 绑定数为 0」即打回重写）而不是信 LLM 自觉——这正是 ReBind「通用 MLLM 写不可靠、要专门管」结论在 Nomi 约束下的等价物（Nomi 不训模型，就用校验闸兜）。
- **为什么进 🟢**：纯 prompt/编排层改动，不碰模型内部、不训练、model-agnostic（P4）；底座（@imageN 单源投影 + 参考槽语义 + 连线在前单源参考口径）全部现成，是「把已有件重新对齐」不是从零造。
- **诚实边界**：① 它的量化提升是在**它家训练的** Instruct+Edit 上测的，「只改 prompt 格式喂 vendor 黑盒模型」能吃到多少收益**没有直接实验**——落地后要用 Nomi 自己的 eval（参考一致性 lane）A/B 验证，别默认等效；② 各 vendor 对 `@imageN` 式指代的理解力不一（Sora2 系确认吃多参考指代，其他待真机验）——按模型能力位分档，不支持多参考的模型不硬塞；③ 项目页未给指令格式的 token 级规范，落地按「语义位置内嵌标签」的思想自定格式即可，别 bluff 说抄了它的格式。

## 🟡 架构教训（影响 Nomi 设计决策，非即插即用）

### 实体/角色记忆第四波收敛 + 强信号：可灵在把「原生多镜 + 多参考」做进模型（SlotMem / EM-Vid / UnityShots / Memento，2026-05~07）

[SlotMem 2607.15772](https://arxiv.org/abs/2607.15772)（2026-07-17，三天新，HKU，[有码](https://github.com/YilaiLiu-HKU/SlotMem) 实测 200）· [EM-Vid 2605.23610](https://arxiv.org/abs/2605.23610)（2026-05-22，Adobe 系作者 Gadelha/Huang，未逐一确认）· [UnityShots 2606.21661](https://arxiv.org/abs/2606.21661)（2026-06-19，作者群含 Xin Tao/Pengfei Wan/Jiaya Jia，可灵系特征，[项目页](https://jackailab.github.io/Projects/UnityShots) 实测 200）· [Memento 2606.14667](https://arxiv.org/abs/2606.14667)（2026-06-12，ernie-research=百度系项目页）。

- **四篇全在模型内部做实体/角色记忆**：SlotMem 用 cross-attention 探针定位角色 token、压成 role-wise slot 记忆再定向注入；EM-Vid 是 training-free 但靠稀疏 token conditioning 限制 self-attention（仍碰采样）；UnityShots 直接在 LTX-2.3 上训双槽记忆（LTM 锚开场镜 + STM 持前镜尾）+ 切点门控；Memento 用「记忆能否重建主体」当训练目标。**结论同前轮：凡碰模型内部的一致性方案 Nomi 一律接不了**，编排层（GroundShot 路线）仍是 Nomi 唯一可走的那条。
- **真正的新信号是产业侧**：UnityShots（可灵系）已经把「一次调用出多镜 + 每镜带参考身份 + 切点强度可调」训成 backbone 并放了 200 序列 benchmark；ReBind（同族作者群）在做多参考归属。**合读 = 可灵在把 Nomi 现在用编排层拼的东西（decompose-stitch + 参考回填）下沉进模型原生能力。**影响 Nomi 哪个决策：P4 能力层要**预留「原生多镜」能力位**——某 vendor 支持「一次调用生成整段多镜（带每镜参考/切点控制）」时，拆镜头执行器应该能整段委托而不是永远逐镜拆发（类似现有 `supportsReferenceImages` 的能力位思路）。现在不用建，但拆镜头执行链路做大改时别把「一镜=一次调用」焊死成不变量。
- **诚实边界**：UnityShots 未放权重（只项目页+benchmark 声明）；「可灵会开放原生多镜 API」是推断不是事实，落成能力位前等真 API 文档（R5）。

### 自然语言运镜理解：VLM（连 Gemini 3.1 Pro）会把平移当旋转、左当右、物动当机动——Nomi 的运镜 VLM 校验不能裸用（2026-07-03，ECCV 2026）

[arxiv 2607.03043](https://arxiv.org/abs/2607.03043) · 作者 Yuwen Tan / Boqing Gong 等 · ECCV 2026 已中 · 二级影视运镜 taxonomy + 原子 benchmark（真实+合成视频）· 微调 VLM-8B 在其 benchmark 上超 Gemini 3.1 Pro 10-11%，但离人类仍有大 gap · 日期：2026-07-03。
- **影响 Nomi 两个决策**：① **shotVerify/评测里凡用 VLM 判「运镜对不对」的轴都不可裸信**——translation↔rotation、left↔right、object-motion↔camera-motion 是实测系统性混淆项，Nomi 若在镜级校验或 eval 里让 VLM 判运镜方向，要么用它的 taxonomy 做原子化提问（别问「运镜对吗」，问「相机在平移还是旋转」二选一分解），要么把运镜轴从 VLM 判词降级成「不判、只人工抽查」；② **反向坐实 Nomi 的 video_ref 运镜路线**——语言→运镜既然连理解都不可靠，「3D 导演台渲小片喂 vendor 全能参考」把运镜从语言通道挪到视频通道，绕开的正是这个坑（记忆 `ai-camera-move-tool-plan` 路线又添一证）。
- **诚实边界**：论文重点是理解侧（训 VLM），Nomi 不训模型；吃的是「失败模式清单 + 原子化提问法」。其 benchmark 若放出可作运镜 judge 的校准集（本轮未见 repo，标待核实）。

### JudgeFit：VLM judge 的评审 rubric 要按 judge 各自的盲区定制，全局统一 rubric 平均差 32%（2026-06-22）

[arxiv 2606.22918](https://arxiv.org/abs/2606.22918) · 作者 Yu Cao / Ziquan Liu / Shaogang Gong（QMUL 系特征，未逐一确认）· 16 个 VLM×8 家族实测 · 日期：2026-06-22。
- **机制**：让目标 VLM 自己枚举它能看出的错→聚类成初始 taxonomy→按人类评分校准、诊断哪些维度它判不可靠/冗余→LLM 修 taxonomy→迭代到收敛。每个 judge 得到自己的「量尺」，held-out 上全员优于全局统一 schema，平均相对提升 ~32%，且暴露各家族的盲区模式。
- **影响 Nomi 哪个决策**：Nomi 的 VLM judge 面（shotVerify 三轴 / 姿势观察器 / evals 生成质量 lane）目前是「一套 rubric 喂任意 judge 模型」。JudgeFit 说这会把 judge 换模型时的可靠性悄悄打折——与记忆 `self-improving-harness-loop` 的「judge 校准铁律」同向但更具体：**换 judge 模型 ≠ 只换 model id，rubric 要随 judge 重校**。近期不必建全套迭代管线，但 evals 里该记录「rubric 是对哪个 judge 校准的」，换 judge 时至少重跑一次标注对齐。
- **诚实边界**：它做的是物理常识域，Nomi 的域（身份/构图/运镜）盲区分布未必同构；无代码仓核实到。

### NEWTON：「规格瓶颈」——prompt 是物理世界的有损压缩，模型再大也补不回没写的参数；agent 该用工具算出显式参数当条件（2026-05-18）

[arxiv 2605.06924→2605.18396](https://arxiv.org/abs/2605.18396) · [项目页 Newton026.github.io/newton](https://Newton026.github.io/newton) · 日期：2026-05-18。
- **核心论断**：VideoPhy-2 上最好的模型 joint accuracy 才 32.6%，根因不是模型弱而是 **specification bottleneck**——文本 prompt 把决定动力学的参数丢了，「no amount of model scaling can recover what was never specified」。解法把视频生成降级成 agent 工具箱里的一个 action：planner 编排物理工具（关键帧生成/科学计算/prompt 精化）构造富条件 + verifier 闭环。
- **影响 Nomi 哪个决策**：给 Nomi 的 3D 导演台 → prompt 链路一个第一性背书：**能算出来的别用形容词**。Nomi 已经在做的方向（焦段 mm、机位参数、站位坐标从 3D 台派生进条件）是对的；反过来审视拆镜头规划师的 prompt 模板——凡「大概/缓慢/近一些」这类可被 3D 台/结构化字段量化的描述，都是规格瓶颈的漏点。与 ReBind 合读：**prompt 的两大漏 = 归属没写显式（ReBind）+ 参数没写显式（NEWTON）**，Nomi 的 prompt 构造层这两刀可以一起补。
- **诚实边界**：物理域结论外推到叙事/身份域是类比不是实验；其 planner 是训练的，Nomi 只吃「显式参数化」原则。

### World Narrative Model：「4D 编排 previz → 渲染」范式宣言，点名「抽卡循环」是工业级生成的死穴——Nomi 动态 previz 押注再添一证（2026-06-30）

[arxiv 2606.31946](https://arxiv.org/abs/2606.31946) · 日期：2026-06-30。
- **主张**：把「渲什么」（结构化 4D 物理叙事：几何/运动/相机/光照，实例级、可量化）与「怎么渲」（像素生成）解耦，用协作 agent 把文本/参考视频/草图翻成 4D pre-visualization 再喂生成——直指现有 T2V「gacha loop（抽卡）」让专业创作不可行。
- **影响 Nomi 哪个决策**：与记忆 `3d-staging-commoditized-pivot-to-dynamic-previz` 的判断完全同向——静态站位已商品化，**动态 previz（4D：走位+运镜+时序）是还没被抄平的层**。学界现在把它当 paradigm 喊出来了，说明这个空窗在收窄：Nomi 的 previz→video_ref 链路（导演台录 take→参考视频→成片）要抓紧从「能用」推到「顺手」，别等 vendor 原生 4D 条件接口出来把这层也商品化。
- **诚实边界**：偏 position/系统宣言，未核实其系统是否可复现；当方向佐证不当实现参照。

## 🔵 对标基准（接进 eval 体系自评 Nomi）

### KeyFrame-Compass：首个关键帧条件生成 benchmark——「首帧/关键帧到底约束住了没」六轴可量化，storyboard 网格喂参考被实测大多失效（2026-07-15）

[arxiv 2607.14202](https://arxiv.org/abs/2607.14202) · 35 页 · 本轮未核实到代码仓 · 386 样本 × 3 应用域 × 4 关键帧密度，9 个代表性视频系统实测 · 日期：2026-07-15。
- **它量的正是 Nomi image-first 模式的立身假设**：Nomi 拆镜头默认「先出每镜首帧图，图转视频继承 first_frame」——整条链路押在「给的关键帧能忠实出现在视频里」。KeyFrame-Compass 把「关键帧执行」拆成六轴（presence 出现没 / fidelity 像不像 / temporal ordering 顺序对不对 / localization 出现在对的时刻没 / persistence 保持住没 / uniqueness 没重复吧）+ MLLM 证据接地的整体质量判，实测发现：**忠实执行关键帧 ↔ 视频自然度是显式 trade-off；关键帧越密退化越狠；大多数开源模型把 storyboard 网格图当整图理解、不当有序帧序列**。
- **怎么接进 eval 体系**：① 给 Nomi 图转视频桥（`convertShotToVideo.ts`）的产物加「首帧忠实度」量化轴——六轴里 presence/fidelity/persistence 三轴对 Nomi 直接适用，接进 `gatherShotVerifyInputs.ts` 的镜级校验当第五轴（转视频镜专属），judge 走现有 VLM 链路；② 首尾帧模型（Seedance 系）用 ordering/localization 两轴验尾帧真落在尾部；③ **警示项**：Nomi 任何「把参考卡/分镜网格整图喂给视频模型当序列指令」的玩法（referenceSheet 类）默认当无效设计——实测大多数模型读不懂网格的时序语义，要传就拆开逐帧传。
- **诚实边界**：无代码仓核实到，六轴要照定义自实现（Nomi 已有 VLM judge 基建，成本在 prompt 工程不在训练）；其 386 样本集未必放出，用 Nomi 自己的拆镜头产物算即可。

### VGIF-Score：把 prompt 解析成时空约束 DAG 逐项 QA + 短路诊断——shotVerify「内容偏差」轴的升级参照（2026-07-15，PRCV 2026）

[arxiv 2607.13527](https://arxiv.org/abs/2607.13527) · [repo github.com/PRIS-CV/VGIF-SCORE](https://github.com/PRIS-CV/VGIF-SCORE)（实测 200，内容未逐项核）· BUPT PRIS 系 · 223 长结构化 prompt / 4.3K 细粒度评测项 / 14 模型 3K+ 视频实测 · 日期：2026-07-15。
- **机制两支**：客观支把 prompt 解析成 **Spatio-Temporal DAG**（原子约束 + 依赖边），做依赖感知 QA——**父约束挂了子约束短路跳过**（省 judge 调用）；主观支用指令条件化 AutoRubric 判运镜/画面纯净/运动平滑/物理。产出统一分 + 逐约束诊断（哪条指令没跟上，不是一个总分）。
- **怎么接进 eval 体系**：Nomi shotVerify 现在的「内容偏差」判定是整体问 VLM；VGIF 的做法是先把镜 prompt 拆成原子约束（谁/在哪/做什么/何时）再逐项 QA + 短路——两个直接好处：① **可归因**（回灌重试时知道具体哪条约束挂了，改 prompt 改得准，正好喂 `storyboardLoopBudget.ts` 的有界重试让每轮重试有的放矢）；② **省额度**（短路跳过依赖失效的子项，VLM 调用次数降）。落点=shotVerify 内容轴的实现方式升级，不加新轴。
- **诚实边界**：repo 存在但内容未核（论文说 code will be available，可能占位）；其 prompt 解析器要适配 Nomi 的镜 prompt 格式，照思想自实现更稳。

### VideoWeaver：拿 Claude Code/Codex 级 agent harness 考「长视频生成」的 benchmark + agent-as-judge 查执行轨迹——Nomi 作为 agent 工作台第一次有同类考卷（2026-06-06）

- **为什么对 Nomi 特殊**：前八轮的 benchmark 都在考「模型/管线」，VideoWeaver 第一次考「**通用 agent harness 能不能自己组装 workflow 做长视频**」——这就是 Nomi 的物种（agent+技能库+画布工具）。三个实测结论都踩在 Nomi 的设计上：① **显式 composition skill > 裸用 foundation skills**（=Nomi playbook/技能库方向的直接背书）；② **skill evolution（按评测反馈精化+合并技能）进一步提质**（=Nomi 记忆 `self-improving-harness-loop` 的方向，且给了具体算法：执行→优化器精化各技能副本→pair-wise merge 成类级技能）；③ agent-as-judge **同时查执行轨迹和成片**（过程指标：规划/技能遵循/参考素材用没用对；产物指标：格式/prompt 达成/音画一致/情节逻辑），与人评对齐度尤其在过程指标上高。
